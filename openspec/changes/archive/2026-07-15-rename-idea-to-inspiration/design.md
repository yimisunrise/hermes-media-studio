## Context

当前应用中对同一内容管理概念使用了两套术语：「思路池/思路」和「灵感」。具体表现为：

- 菜单/面板标题使用「思路池」
- 弹窗、按钮、空状态、错误日志中混用「思路」和「灵感」
- DB schema label 使用了「思路描述」

改动范围全部为前端 UI 文本，不涉及数据层、API 层或内部命名。

## Goals / Non-Goals

**Goals:**
- 所有面向用户的 UI 文本中「思路池」→「灵感」
- 所有面向用户的 UI 文本中「思路」→「灵感」
- 保持英文内部命名不变（IdeaBoard, idea.title, ideaId 等）

**Non-Goals:**
- 不改文件名、类名、变量名、函数名
- 不改 API 路由或数据库表名/字段名
- 不改数据存储内容

## Decisions

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | 面板标题「思路池」 | 「灵感」 | 用户明确要求，直接将池的概念去掉，简洁直观 |
| 2 | 空状态「暂无思路，...」 | 「暂无灵感，在顶部的输入框中记录第一个灵感吧」 | 统一术语 |
| 3 | 错误日志「创建思路失败」 | 「创建灵感失败」 | 虽然是 console.error，但保持一致便于调试时理解 |
| 4 | DB schema label「思路描述」 | 「灵感描述」 | schema label 仅在代码中作为注释性存在，不持久化 |
| 5 | TopicBoard 中的「思路池」引用 | 改为直接指向「灵感」 | 「暂无选题，从思路池选择一个灵感转为选题」→「暂无选题，先去「灵感」中创建灵感，再转为选题」 |

## Risks / Trade-offs

- **低风险**：纯字符串替换，无业务逻辑变更
- **唯一隐含依赖**：如果用户或系统在未改动的文件中引用了「思路」字样的输出，需确认未被遗漏
- **无需迁移**：数据中不存储「思路」相关用户可见文案
