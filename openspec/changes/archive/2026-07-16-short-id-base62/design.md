## Context

当前项目中所有数据库记录的主键通过 `DataRepository.newId()` 和 `meta.uuid()` 使用 `crypto.randomUUID()` 生成标准的 UUID v4（36 位字符串）。此外，`TaskDetail.js`、`AssetGallery.js`、`AgentTaskPoller.js` 中也有多处内联的 `crypto.randomUUID()` 调用。

这些 UUID 在 URL、日志、调试输出和数据库管理中难以阅读和处理。项目处于开发早期，数据量预期在万级以下，单用户/单进程场景，无需分布式 ID 生成。

## Goals / Non-Goals

**Goals:**
- 将 ID 格式从 36 位 UUID v4 替换为 **7 位 Base62 编码的毫秒时间戳**
- ID 单调递增且按创建时间有序
- 无需维护持久化计数器或状态文件
- 零依赖、纯浏览器原生 API 实现
- 与现有 UUID 格式数据共存，不破坏已有记录

**Non-Goals:**
- 不修改表结构 schema 定义（`type: 'uuid'` 标记保持不变）
- 不为旧数据做迁移（新旧 ID 自然共存）
- 不实现分布式 ID 生成（当前无此需求）

## Decisions

### 方案选择：7 位 Base62 毫秒时间戳

选择 Base62（`0-9A-Za-z`）而非 Base36（`0-9a-z`），因为：
- Base62 无大小写歧义，62 字符集密度更高
- 7 位 Base62 可编码 62⁷ ≈ 3.52×10¹² 个值，覆盖直到 2081 年
- 当前毫秒时间戳 ~1.78×10¹²，恰好 7 位填满，没有浪费

### 碰撞处理

同一毫秒内创建多条记录时可能碰撞。处理策略：
- `DataRepository.newId()` 在写入前检查 `repo.get(id)` 是否已存在
- 如果碰撞，忙等 1ms 后重试（时间戳自然推进到下一毫秒）
- 在单用户/单进程场景下，碰撞概率极低，忙等开销可忽略

### 字符集

```
0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz
```

注意：Base62 是大写字母在前，这意味着大写字母的 ID 会比小写字母的 ID 更早出现，排序与时间顺序一致。

### 覆盖范围

当前毫秒时间戳 `1,784,187,xxx,xxx` 编码为 7 位 Base62。62⁷ = 3,521,614,606,208 对应的时间点为 2081-08-05，约 55 年后。届时如项目仍在使用，可扩展至 8 位或更换方案。

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 2081 年溢出 | ID 长度需增加 | 55 年时间窗口，届时方案迭代 |
| 同一毫秒高并发碰撞 | 写入延迟 | `get()` 检查 + 1ms 重试，单用户场景几乎不会触发 |
| 日期信息泄露 | ID 可反解时间戳 | 内容管理系统非安全敏感场景，可接受 |
| 区分大小写 | 人工转录易错 | Base62 已是最紧凑的字母数字方案；可考虑排除易混淆字符（如 0/O、1/l）但会增加长度 |
